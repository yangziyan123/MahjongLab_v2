# MahjongLab 按时间划分的开发计划

## 1. 计划前提

- 开发顺序必须是：`先做复盘需求，再做对战需求`
- 平台内部统一牌谱标准：`mjai JSONL`
- 四个参考仓库的使用定位如下：
  - `Mortal`：AI 决策与复盘内核
  - `mjai.app`：`mjai` 协议、离线模拟、验证工具
  - `mjai-reviewer`：复盘流程与日志转换参考实现
  - `原型设计`：前端页面骨架、视觉样式和交互流程参考

## 1.1 文档与验收规则

- 每个阶段结束后，必须新增一份对应的验收报告
- 验收报告至少包含：
  - 完成了哪些
  - 遗留的问题
  - 验收成果
- 项目所有文档都默认允许修改，不做“冻结后不可改”处理
- 旧文档不做平行分叉维护，优先直接更新当前版本

- `Alembic`
- `Celery`
- `Uvicorn`

### 2.3 数据与基础设施

- `PostgreSQL 16`
- `Redis 7`
- `MinIO / S3`
- `Docker Compose`
- `GitHub Actions`
- `Nginx`
- `Prometheus + Grafana`
- `Sentry`

### 2.4 麻将与 AI 能力栈

- `Mortal`
- `libriichi`
- `mjai.app`
- `mjai-reviewer/convlog`
- `mjai protocol`

### 2.5 为什么这样选

- 复盘、AI、任务编排都和 Python 生态更贴近，后端统一用 `FastAPI + Celery`
- 目录里已有可运行的 `原型设计` 仓库，技术栈是 `Vite + React Router + Tailwind`
- 前端直接沿用该技术方向，能最大化复用已有页面和组件，避免无意义迁移
- `PostgreSQL + Redis + S3` 足够覆盖日志、任务、结果文件和历史数据
- 先把平台核心问题统一到 `mjai` 事件流，后续对战和复盘就不会裂成两套系统

### 2.6 原型仓库复用策略

`原型设计` 当前已具备这些页面骨架：

- `/`
- `/review/import`
- `/review/task/:taskId`
- `/review/report/:reportId`
- `/review/history`
- `/play/config`
- `/play/game/:roomId`
- `/play/result/:sessionId`
- `/play/history`

复用原则：

- 复盘阶段优先复用 `review` 相关页面
- 对战阶段再复用 `play` 相关页面
- 复用页面结构、样式、组件和文案布局
- 不复用 mock 数据和临时页面逻辑
- 接口层、状态管理、鉴权、WebSocket 全部按主项目重新接入

## 4. 阶段 0：项目基线（第 1 周）

### 目标

把项目底座搭起来，并先证明“复盘链路能跑通”。

### 任务

#### 架构与仓库

- 创建主仓库，并把参考源码目录统一纳入主仓管理
- 建立目录结构：
  - `apps/web`
  - `services/api`
  - `services/review-worker`
  - `services/ai-gateway`
  - `services/match-service`
  - `Mortal`
  - `mjai.app`
  - `mjai-reviewer`
  - `原型设计`
- 建立 `.editorconfig`、`ruff/black`、`eslint/prettier`、`commit hooks`

#### 环境与基础设施

- 用 `docker-compose` 启动 `PostgreSQL`、`Redis`、`MinIO`
- 初始化 `FastAPI` 项目和 `Vite + React` 项目
- 初始化 `Alembic` 迁移
- 接入基础日志输出和错误上报

#### 协议与数据

- 确定平台内部唯一日志标准为 `mjai JSONL`
- 定义第一版数据表：
  - `users`
  - `matches`
  - `match_events`
  - `review_jobs`
  - `reviews`
  - `review_entries`
- 定义第一版 API 契约：
  - `POST /reviews`
  - `GET /reviews/{id}`
  - `GET /reviews/{id}/entries`

#### 上游能力验证

- 跑通 `Mortal` 普通模式
- 跑通 `Mortal review mode`
- 跑通 `mjai.app` 模拟器
- 跑通 `mjai-reviewer` 的样例牌谱复盘
- 跑通 `原型设计` 前端原型
- 盘点可直接复用的页面、组件和样式 token

### 交付物

- 主仓库骨架
- 本地开发环境
- 数据库初版迁移
- 架构图 v1
- `Review JSON Schema v1`
- 前端原型复用清单

### 验收标准

- 新机器可在 30 分钟内启动本地开发环境
- 能对一份样例牌谱生成完整复盘 JSON
- `Mortal`、`mjai.app`、`mjai-reviewer`、`原型设计` 都有可复现的运行文档

## 5. 阶段 1：复盘后端 MVP（第 2-3 周）

### 目标

先把“上传牌谱 -> 后台分析 -> 取回结构化复盘结果”的后端链路做出来。

### 任务

#### Review Service

- 建立 `review-service` 和 `review-worker`
- 用 `Celery + Redis` 实现异步任务队列
- 封装 `Mortal review mode` 调用器
- 统一 Review 结果结构，字段至少包含：
  - `kyoku`
  - `honba`
  - `junme`
  - `tiles_left`
  - `actual_action`
  - `expected_action`
  - `details`
  - `shanten`
  - `at_furiten`
  - `rating`

#### 输入适配

- 支持平台内部 `mjai` 日志直接复盘
- 接入 `mjai-reviewer/convlog`，支持 `Tenhou -> mjai`
- 为后续 `Majsoul -> mjai` 预留 Adapter 接口

#### API

- `POST /reviews`：提交复盘任务
- `GET /reviews/{id}`：查看任务状态和摘要
- `GET /reviews/{id}/entries`：分页查看逐巡复盘结果
- `GET /reviews`：复盘列表

#### 数据持久化

- 复盘摘要入库到 `reviews`
- 逐巡结果入库到 `review_entries`
- 原始牌谱与完整 JSON 输出落到 `MinIO`

#### 权限与用户

- 增加最小可用登录能力
- 复盘结果与用户绑定

### 交付物

- 复盘后端服务
- 异步复盘任务链路
- 第一版复盘 API
- 第一版用户体系

### 验收标准

- 内部 `mjai` 牌谱可提交复盘并生成结果
- `Tenhou` 牌谱可转换并进入相同复盘链路
- 复盘任务失败时可重试
- 复盘结果刷新页面后仍可读取
- 至少 20 份样例牌谱通过复盘后端回归测试

### 阶段 1 实施注记（2026-04-14）

- 当前已落地一版“本地 MVP 形态”的复盘后端，代码位于 `services/api`
- 已实装：
  - `FastAPI + SQLAlchemy`
  - `inline_json / upload_file / internal_match`
  - `review-jobs / reviews / uploads / dashboard / me`
  - 本地异步执行器
- 当前未实装：
  - `Tenhou / Majsoul` 外部牌谱适配
  - 独立 `review-worker`
  - `PostgreSQL + Redis + MinIO` 目标部署形态
- 阶段 1 的正式验收结果见：
  - [docs/acceptance/PHASE_1_ACCEPTANCE.md](/d:/Code/MahjongLab/MahjongLab/docs/acceptance/PHASE_1_ACCEPTANCE.md)

## 6. 阶段 2：复盘前端 MVP（第 4-5 周）

### 目标

把复盘结果真正做成可用产品，而不是只停留在 JSON。

### 任务

#### 页面

- 基于 `原型设计` 迁移并接好以下页面：
  - 登录页
  - 复盘上传页
  - 复盘任务页
  - 复盘列表页
  - 复盘详情页

#### 复盘详情页核心模块

- 局列表
- 巡目时间轴
- 当前巡动作对比卡片
- Top 候选动作列表
- 牌局信息卡片
- 错误巡目筛选器

#### 前端交互

- 支持按局切换
- 支持按巡目跳转
- 支持“只看失误巡目”
- 支持根据动作类型筛选：
  - 打牌
  - 立直
  - 鸣牌
  - 和牌 / 流局判断

#### 数据接入

- 用 `TanStack Query` 管理异步拉取与缓存
- 用 `Zustand` 管理当前局、巡、筛选状态
- 清理原型页面中的 mock 数据和本地跳转逻辑
- 接入真实复盘 API

### 交付物

- 可用的复盘 Web 界面
- 复盘上传与查询闭环
- 原型页面到真实业务页面的第一轮迁移

### 验收标准

- 用户可上传牌谱并在 Web 上查看复盘
- 单份复盘可按局、巡目、动作类型切换
- 用户能明确看到“实际动作”和“推荐动作”的差异
- 复盘页在桌面端和移动端都能正常使用
- 至少 10 份真实样例由产品视角完成人工验收

- 当前未实装：

## 7. 阶段 3：牌谱导入与训练入口（第 6-7 周）

### 目标

把复盘产品从“查看报告”升级到“可沉淀训练数据”。

### 任务

#### 导入能力

- 完成 `Tenhou` 导入正式版本
- 增加 `Majsoul` 导入方案 v1
  - 如果不能稳定自动抓取，则先支持用户上传中间导出文件
- 导入失败时提供可读错误信息

#### 训练入口

- 为复盘项增加“加入错题库”
- 为复盘项增加标签：
  - 进攻判断
  - 防守判断
  - 立直判断
  - 鸣牌判断
  - 牌效率
- 支持“我的错题”列表页

#### 统计摘要

- 增加复盘摘要指标：
  - 总失误数
  - 大失误数
  - 立直判断偏差数
  - 鸣牌判断偏差数
  - 防守失误数

### 交付物

- 牌谱导入页
- 错题库页
- 复盘摘要统计

### 验收标准

- `Tenhou` 导入成功率达到内测要求
- 导入失败时用户能看懂失败原因
- 用户能把复盘巡目加入错题库
- 用户能从错题库重新进入对应复盘点

## 8. 阶段 4：复盘链路加固（第 8 周）

### 目标

在进入对战开发前，把复盘产品先做成一个可独立上线的 Beta。

### 任务

#### 性能与稳定性

- 增加复盘任务超时控制
- 增加任务取消、失败重试、死任务扫描
- 限制单用户同时复盘任务数
- 增加 Worker 健康检查

#### 质量

- 建立固定回归牌谱集
- 增加 Review JSON Schema 校验测试
- 增加 API 集成测试
- 增加前端 E2E 测试

#### 观测

- 接入 `Prometheus` 指标：
  - 任务耗时
  - 任务失败率
  - Worker 重启次数
- 接入 `Sentry`

### 交付物

- Beta 级复盘系统
- 回归测试资产
- 监控与报警

### 验收标准

- 复盘任务成功率达到预期
- Worker 崩溃后可自动恢复
- 至少 50 份样例牌谱在回归集上稳定通过
- 复盘功能可独立向外演示

## 9. 阶段 5：对战后端 MVP（第 9-10 周）

### 目标

在复盘链路稳定后，再开始做服务端权威的对战引擎。

### 任务

#### Match Service

- 建立一桌牌局的状态机
- 服务端维护权威状态：
  - 手牌
  - 牌山
  - 河
  - 副露
  - 立直
  - 供托
  - 局进程
- 用户动作统一走服务端合法性校验

#### AI Gateway

- 封装 `Mortal` 实时决策调用
- 建立长生命周期 Worker 池
- 支持 AI 超时、崩溃、重启
- 预留 AI 难度档位参数

#### 日志与持久化

- 每桌对局完整记录为 `mjai JSONL`
- 对局结束写入 `matches` 与 `match_events`
- 为后续自动复盘预留任务钩子

#### API / WebSocket

- `POST /matches`
- `GET /matches/{id}`
- `POST /matches/{id}/actions`
- `WS /matches/{id}`

### 交付物

- 对战后端
- AI Gateway
- 对局日志链路

### 验收标准

- 通过接口可创建一场 `1 人 + 3 AI` 对局
- 不依赖前端，后端可完成一场完整半庄
- 每场对局都能生成完整 `mjai` 日志
- AI Worker 异常退出时能自动重启或安全失败

## 11. 阶段 7：对战与复盘闭环打通（第 13-14 周）

  - 对局结算页

- 增加“本场最值得复盘的 3 个节点”

### 交付物

- 对战后自动复盘
- 对局历史闭环
- 训练入口闭环

### 验收标准

- 任意一场内部对局结束后都能自动进入复盘流水线
- 用户能从历史页回到牌局和复盘结果
- 至少一场完整对局能完成：
  - 开局
  - 对战
  - 结算
